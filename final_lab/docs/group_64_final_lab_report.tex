\subsubsection*{Objective}
In this experiment, we will compare both \texttt{dijkstra\_approx()} and \texttt{bellman\_ford\_approx()} to \texttt{dijkstra()} and \texttt{bellman\_ford()} respectively, with increasing values of $k$. This experiment was chosen in order to test if the approximation algorithms converge to their true counterparts when increasing $k$ of the approximation algorithms, and at what rate these changes occur at. The conditions for the experiment are outlined as follows:
\begin{itemize}
    \item Generate a sparse graph of 1000 nodes, with edge weights between 1 and 100
    \item Each edge should be generated with a probability of 5\% 
    \item Values of $k$ will increase from 1 to 20, with a step size of 1
    \item For each value of $k$, we will calculate the total distance error between the approximation algorithms and the true algorithms.
\end{itemize}
\subsubsection*{Results}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{figures/Figure_1.png}
    \caption{Distance Error of Approximation Algorithms with Increasing $k$}
\end{figure}

\newpage

\subsubsection*{Discussion}
Observing the graph, we can see that as $k$ increases, the average node distance error of both approximation algorithms decreases. However, this approximation error swiftly decreases, and plateaus by $k=8$ for Dijkstra, and $k=5$ for Bellman-Ford. 

Considering the Dijkstra's approximation algorithm, this plateau is likely due to the fact that the approximation algorithm functions by performing Dijkstra's algorithm on the subgraph of the $k$ nearest neighbours of the current node being iterated through. This means that as $k$ increases, the approximation algorithm considers a much larger portion of the graph, therefore resulting in a much closer approximation to the true Dijkstra's algorithm. 

Observing the Bellman-Ford approximation algorithm, this plateau is likely due to the nature of how the algorithm functions. The Bellman-Ford approximation algorithm functions by searching for the shortest path from the source node to the target using only $k$ edges, with $k$ increasing iteratively. Therefore on a sparse graph, as there are not many edges, the shortest path from the source to the target node is likely to be found within a very small number of edges, resulting in a very early plateau in the approximation error.

\subsubsection*{Objective}
In this experiment, we will compare both \texttt{dijkstra\_approx()} and \texttt{bellman\_ford\_approx()} to \texttt{dijkstra()} and \texttt{bellman\_ford()} respectively, with increasing graph density, and fixed values of $k$. The conditions for the experiment are outlined as follows:
\begin{itemize}
    \item Graph density will increase from 1\% to 10\%, with a step size of 1\% each iteration
    \item For each graph density, generate a sparse graph with 500 nodes, with edge weights between 1 and 10
    \item Values of $k$ will be 1, 3 and 5
    \item For each graph density and value of $k$, we will calculate the total distance error between the approximate and true algorithms
\end{itemize}
\subsubsection*{Results}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{figures/Figure_2.png}
    \caption{Distance Error of Approximation Algorithms with Increasing Density}
\end{figure}
\subsubsection*{Discussion}
Observing the graph, we can see that as graph density increases for all values of $k$, the distance error for the Bellman-Ford approximation decreases, while the distance error for Dijkstra's approximation increases.

The reason for which the distance error for the Bellman-Ford approximation algorithm decreases is likely due to the fact that, as there are more edges present in the graph, there is more likely to be a more direct connection between two nodes. This increase in density within the graph means that the shortest path between nodes becomes shorter.

The reason for which the distance error for Dijkstra's approximation algorithm increases is likely due to how the approximation algorithm functions. The approximation function for Dijkstra's is greedy. When a graph is less dense, each node has less edges, meaning that there are less candidate paths to explore, which will allow us to stay below our $k$ value. However, when a graph is much more dense, there are many more possible paths to check, meaning that we are likely to reach our threshold of relaxing earlier. This prevents us from finding more efficient paths, as a node cannot be updated once it reaches the threshold of $k$. 

Ultimately, when we are limiting $k$ for each algorithm, the Bellman-Ford algorithm seems to perform better in more dense graphs, as the Total Distance Error is substantially lower than that of the Dijkstra approximation.

    \item For each difference between the true and approximate error, we will normalize it with the number of nodes in the graph
\end{itemize} 
\subsubsection*{Results}
\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{figures/Figure_3.png}

\subsubsection*{Discussion}
Observing the graph, it is evident that as the number of nodes increases, the relative distance error of the Bellman-Ford approximation algorithm decreases, while that of the Dijkstra approximation increases.

The reason for which the relative distance error of the Bellman-Ford approximation algorithm decreases is likely for the same reason that it decreases when increasing graph density. As the graph density remains at 5\%, when the number of nodes increase, the number of edges also greatly increase. This means that larger graphs become very well connected, and as a result, shortest paths between nodes do not become much larger. This means that nodes are still reachable in few steps, resulting in the graph size not hindering the Bellman-Ford approximation algorithm.

The increasing number of edges for the graph is also why the relative distance error for Dijkstra's approximation increases on larger graphs. When graphs are smaller, Dijkstra's approximation algorithm is able to exhaust a larger relative portion of the graph before reaching the limits given for the function. However on larger graphs, as there are more nodes and edges, it will reach these limits before it is able to explore a substantial relative portion of the graph. This means that only a small portion of the distance in a route can be relaxed, which becomes less substantial when the graph, and subsequently paths, are longer.

\subsubsection*{How A* works}
\addcontentsline{toc}{subsubsection}{How A* works}
The A* Algorithm is an algorithm used find the shortest path from a start node to a target node in a directed weighted graph. Its algorithm is practically the same as Dijkstra's, the only difference is how it determines priority (the key value) in the priority queue.
\\

In A*, the key value used in the priority queue is equal to the total weight along the path traversed (like Dijkstra's), plus some value coming from a heuristic function. This heurisitc function outputs a number relative to its distance from the destination, guiding the algorithm to explore nodes that are closer to the destination and more likley to be along the shortest path.

\subsubsection*{Issues A* addresses}
\addcontentsline{toc}{subsubsection}{Issues A* addresses}
A* addresses a problem that Dijkstra has where it explores nodes that are clearly not on the shortest path. For example, if we were to calculate the shortest route from Toronto to Hamilton with Dijktra's it'll probably venture out to Scarborough before finding Hamilton, which is obviously inefficient. If we were to use A*, we can use a heurisitc function that calculates the straight line distance to Hamilton. This heurisitc function would make A* prioritize exploring western locations relative to Toronto, aside from exploring Scarborough. This would result in the shortest path being found much quicker.

\subsubsection*{Empirically testing A* against Dijkstra's algorithm}
\addcontentsline{toc}{subsubsection}{Empirically testing A* against Dijkstra's algorithm}
You could empirically test the speed difference between A* and Dijkstra's algorithm by generating varying number of points in 2D space, with random edges going between the two and running the algorithm on random starting and target nodes, comparing the execution times. It is important to note that for any node, its heuristic value must be a lower bound to the total weight of the path to the target node, otherwise A* could break and return a path that isn't the shortest.

\subsubsection*{Effects of an arbitrary heuristic function}
\addcontentsline{toc}{subsubsection}{Effects of an arbitrary heuristic function}
If you generated an arbitrary heuristic function, A* could return the wrong result if the function isn't a lower bound of the total weight of the path to the target node. An arbitrary heuristic function could also push the algorithm to prioritize paths that aren't close to the shortest, which can take a lot longer then Dijktra's algorithm.

\subsubsection*{Applications where A* should be used instead of Dijkstra’s algorithm}
\addcontentsline{toc}{subsubsection}{Applications where A* should be used instead of Dijkstra’s algorithm}
A* should be used in applications where there exists a heurstic function that is quick to compute, is always a lower bound to the weight of the shortest path from the inputted node to the target node, and the function decreases as you move to nodes closer to the target. Because straight line distances are always the lower bound to the shortest route, they are easy to calculate ($O(1)$ via pythagorean theorem if you have cartesian coordinates) and they decrease as you get closer to the target node. This is why A* is so good for shortest route calculations.

% A* is used in apps such as google maps for calculating the shortest route to a destination. In scenario's like shortest route calculators, the nodes of the graph would be intersections with their distances between them being the weights of the edges between them, and the heuristic function would return the straight line distance between the inputted node to the targeted node.

\subsection*{Data and Graph Construction}
\addcontentsline{toc}{subsection}{Data and Graph Construction}
We built a weighted graph from the London Tube data. Each station was treated as a node, and each 
connection between two stations was treated as an edge. Using \texttt{london\_connections.csv}, we added 
an edge whenever two stations were connected in the dataset. The weight of each edge was based on the 
geographical distance between the two stations, which we computed using the latitude and longitude 
values in \texttt{london\_stations.csv}. In our implementation, these connections were treated as bidirectional.

This gave us a real transportation network to work with instead of a random graph. Because the London Tube 
has both physical layout and line transfer structure, it was a good dataset for comparing Dijkstra's 
algorithm and A* in a meaningful way.

\subsection*{Heuristic Function}
\addcontentsline{toc}{subsection}{Heuristic Function}
For each destination station \(d\), we built a heuristic dictionary for A*. The heuristic value of a node 
was the straight line distance from that node to \(d\). This matches the suggestion in the project 
description to use the physical ``as the crow flies'' distance as the heuristic.

This is a reasonable heuristic for the London Tube graph because every station has real coordinates. A station 
that is closer to the destination in physical distance is often a better direction to explore. Because of 
this, A* gets useful guidance toward the goal, while Dijkstra does not use this extra information.

\subsection*{Experimental Setup}
\addcontentsline{toc}{subsection}{Experimental Setup}
We compared Dijkstra and A* on many source destination pairs in the London Tube graph. For each pair, we 
recorded the runtime of both algorithms, rebuilt the shortest path, measured the path length, and counted 
the minimum number of line transfers on that path. We grouped the journeys into three categories:
\begin{itemize}
    \item \textbf{Same line}: \(0\) transfers,
    \item \textbf{One transfer}: \(1\) transfer,
    \item \textbf{Several transfers}: \(2\) or more transfers.
\end{itemize}

Before studying the performance results, we checked that the implementation was correct. Our test suite showed 
that the London data loaded correctly and that A* matched Dijkstra on sampled and additional 
source destination pairs. After generating the full \texttt{part3\_results.csv} file, we also checked the 
entire output. This check found \(0\) distance mismatches between Dijkstra and A*, and \(0\) invalid rows 
with infinite distance or empty paths. Therefore, the results below are based on correct shortest path outputs.

\subsection*{Results}
\addcontentsline{toc}{subsection}{Results}
Figure~\ref{fig:part3-path-length} shows the median runtime of both algorithms grouped by shortest path length. 
In general, the median runtime increases as the path length increases. However, A* stays faster than Dijkstra 
across almost the whole range of path lengths. This shows that the geographical heuristic is still useful even 
for longer journeys.

Figure~\ref{fig:part3-transfer-category} shows the median runtime grouped by transfer category. Same line trips 
are the fastest for both algorithms, trips with one transfer are slower, and trips with several transfers are 
the slowest. In all three categories, A* has a lower median runtime than Dijkstra.

Figure~\ref{fig:part3-speedup} shows the speedup of A* over Dijkstra, where
\[
\text{speedup} = \frac{\text{Dijkstra runtime}}{\text{A* runtime}}.
\]
The median speedup is about \(1.3\times\) for same line trips, about \(1.8\times\) for trips with one transfer, 
and about \(1.75\times\) for trips with several transfers. This means that A* is faster in all three categories, 
and its advantage becomes larger when the trip is less direct.

We also looked at runtime grouped by the exact number of transfers. The same pattern appeared there too: as 
the number of transfers increased, the runtime of both algorithms usually increased, but A* still stayed faster 
than Dijkstra.

\subsection*{Discussion}
\addcontentsline{toc}{subsection}{Discussion}
Overall, our experiments show that A* was consistently faster than Dijkstra on the London Tube graph. The gap 
between the two algorithms was smallest for short same line trips, where both algorithms had less work to do. 
These were the cases where the two algorithms were the most similar. Even in these simpler cases, A* still had 
an advantage.

As trips became longer or needed more transfers, the difference between the two algorithms became larger. The 
main reason is the geographical heuristic used by A*. Since the graph has real coordinate data, the straight line 
distance to the destination gives a meaningful estimate of how far a node is from the goal. This helps A* move 
in the right direction, while Dijkstra explores the graph more evenly and may spend more time on nodes that do 
not help reach the destination quickly.

The transfe based results support this idea. Same line trips are usually simpler and have fewer possible route 
choices. Trips with one or several transfers are more complicated and create more possible intermediate routes. 
In these harder cases, the heuristic becomes more useful because it helps A* avoid wasting time on less relevant 
parts of the graph. This matches the speedup values in Figure~\ref{fig:part3-speedup}, where A* shows a bigger 
advantage once the trip is no longer a direct same line journey.

The path length results show the same idea in another way. Shorter paths are usually easier because the 
algorithms can finish quickly. As the path length grows, the search space becomes larger and there are more 
choices to consider. Because of this, the median runtime of both algorithms increases. However, the increase 
is much larger for Dijkstra than for A*. This suggests that the heuristic becomes more useful as the trip 
becomes longer and less direct.

In our experiments, Dijkstra did not show strong evidence of being better than A* on typical journeys. The 
closest cases were short same line trips, where their runtimes were the most similar. For this dataset, 
A* appears to be the better practical choice when the goal is to find a path to one specific destination and a 
useful heuristic is available.

We included the line based part of the assignment by using the transfer count of the rebuilt shortest path. We 
treated \(0\) transfers as a same line trip, \(1\) transfer as a one transfer trip, and \(2\) or more transfers 
as a trip with several line changes. This helped us connect the algorithm results directly to the structure of 
the London Tube system instead of only describing the graph in a general way.

\subsection*{Limitations}
\addcontentsline{toc}{subsection}{Limitations}
One limitation of this experiment is that the edge weights were based on geographical distance instead of real 
travel time. Because of this, the graph does not include waiting time, different train speeds, station delays, 
or walking time during transfers. In the same way, the heuristic only uses straight line distance and does not 
directly include transfer cost. Even with these simplifications, the heuristic still gave A* a clear advantage 
on this dataset.

\subsection*{Conclusion}
\addcontentsline{toc}{subsection}{Conclusion}
In conclusion, the London Tube experiments show that A* is usually the better choice when a useful heuristic is 
available. On this dataset, A* had lower median runtime than Dijkstra across different path lengths and across 
different transfer categories. The difference was smallest for simple same line trips and larger for trips with 
one or several transfers. These results support the idea that A* is especially useful on real transportation 
networks where a good lower bound heuristic can be computed easily.

\subsubsection*{Figure 2's design principles and patterns}
\addcontentsline{toc}{subsubsection}{Figure 2's design principles and patterns}
In Figure 2, the design principles and patterns being used are:
\begin{itemize}
  \item \textbf{Strategy and Template Design Pattern:} The SPAlgorithm is a template strategy for the strategies implementing the following algorithms: Dijkstra's, Bellman Ford, and A* 
  \item \textbf{Adapter Design Pattern:} In the assignment instructions, A*'s strategy is supposed to be adapted from its implementation in \textbf{Part 2}. I also decided to adapt \textbf{Bellman\_Ford} from its implementation in \textbf{Part 1}
  \item \textbf{Abstraction Design Princple:} The \textbf{ShortPathFinder} Class is abstracted from the implementation of the sorting algorithms and the graphs. This is because their innerworkings are encapsulated within their class and hidden from \textbf{ShortPathFinder}
\end{itemize}

\subsubsection*{Robustness of the future implementation's of different node representation}
\addcontentsline{toc}{subsubsection}{Robustness of the future implementation's of different node representation}
If in the future, we may need nodes to be represented as something other then integers, we can implement a class that is a decorator to \textbf{Graph}. All this new class would have is an instance of \textbf{Graph} or its children, and a dictionary that maps each node's in-graph integer representation to whatever you want the node represented as. You can then add a method to get the \textbf{Graph} instance to do shortest path calculations, and decorator methods to \textbf{Graph} (e.g. add nodes with one function call giving a name as a string, which adds an integer to the \textbf{Graph} instance and a mapping to the name in the dictionary). 
\\

This solution abstracts what nodes represent in a graph away from \textbf{Graph} and \textbf{SPAlgorithm}, keeping implementations of \textbf{Graph} and \textbf{SPAlgorithm} simple and avoiding the need to change their implementations.
\\

\subsubsection*{Potential graphs that can be implemented}
\addcontentsline{toc}{subsubsection}{Potential Graph's that can be implemented}
We can implement many other graphs such as:
\begin{itemize}
    \item Directed Acyclic Graphs and Bipartite Graphs by adding a method that returns a boolean if it follows their rules and/or preventing changes from being made that break the rules. 
    \item Undirected graph's by changing "add\_edge" to add two directed edges going both ways. 
    \item Multigraphs which can have multiple edges between two nodes. One way is if you want duplicate edges with different weights, the "w" method would need to return a list of floats. We would also need to update the "add\_edge" method and how the weights are stored to support this. 
\end{itemize}
